# 正式章节创作模板

_用于《地球项目事故调查报告》每一章从动笔前检查到发布归档；模板本身不是小说正文。_

---

## 📋 章节基本信息

| 字段 | 填写内容 |
|---|---|
| 章节编号 | 第 `XXX` 章 |
| 章节标题 | 待填 |
| 所属卷 | 第 `X` 卷 |
| 目标字数 | `3000—4000` 字；按章节卡细化 |
| 剧情时间 | 地球日 `DX`；必要时补舰上标准日 |
| 主要地点 | 最多列 3 个 |
| 视角人物 | 默认陈默；改变视角必须说明原因 |
| 本章主任务 | 只写一个可验证的核心推进 |
| 章尾钩子 | 必须自然引出下一章 |

## 🎯 动笔前输入

开始写作前必须同时打开并核对：

- `docs/章节规划.md` 中的本章任务
- 对应卷的详细场景卡
- `docs/连续性台账.md` 中的上一章结束状态
- `docs/时间线.md` 中的日期和倒计时
- `docs/人物状态.md` 中的在场人物状态
- `docs/伏笔表.md` 中计划埋设、加深或回收的条目
- `docs/世界观规则.md` 中本章会触及的权限和技术规则

若这些文件互相冲突，先修正文档，再开始写正文。

## ✍️ 场景设计表

每章建议安排 4—6 个场景。下表中的字数是正文预算，不是必须机械凑数。

| 场景 | 目标字数 | 角色目标 | 阻力 | 专业/线索动作 | 场景结束变化 |
|---|---:|---|---|---|---|
| 1. 开场承接 | 待填 | 待填 | 待填 | 待填 | 待填 |
| 2. 问题升级 | 待填 | 待填 | 待填 | 待填 | 待填 |
| 3. 判断或行动 | 待填 | 待填 | 待填 | 待填 | 待填 |
| 4. 结果与代价 | 待填 | 待填 | 待填 | 待填 | 待填 |
| 5. 章尾钩子 | 待填 | 待填 | 待填 | 待填 | 待填 |

每个场景至少发生一种变化：信息变化、风险变化、关系变化、资源变化或人物决定变化。只重复讨论而没有变化的场景应删除或合并。

## 🧭 叙事与人物检查

### 陈默

- 外在表达谨慎、不擅长当面拒绝；内心吐槽可以更直接
- 遇到异常先排除常规原因、保存证据，再提出判断
- 可以紧张和犹豫，但不能丢失基本工程能力
- 爽点来自定义问题、控制边界和承担有限结论，不靠气势碾压

### 芙娅及配角

- 芙娅有独立职责和判断，不能只作为提示工具或感情奖励
- 赵启明可以催进度和甩流程，但关键时刻必须具备基本负责人能力
- 专业配角必须在自己擅长的领域解决真实问题
- 反派线索通过行为、权限和证据出现，不靠人物主动自曝

### 视角边界

- 正文采用第三人称限知，主要跟随陈默
- 不直接叙述陈默不可能知道的赛珞、洛岑等隐藏状态
- 作者已知、人物已知和读者已知必须在连续性台账中区分

## 🔧 专业内容检查

每章最多重点解释 1—2 个专业概念，并按以下顺序呈现：

1. 先让问题影响人物或现场
2. 再用日常类比说明直觉
3. 最后给出必要术语
4. 让专业判断改变行动方案或证据结论

专业段落若删除后不影响剧情、笑点、线索或解法，说明它只是知识展示，应当删减。

## 💬 笑点与严肃场景

- 日常和流程场景可保持较密笑点，建议每 600—900 字出现一次自然喜剧落点
- 危险处置中，笑点只能来自制度或设备的荒诞反差，不能嘲弄伤者
- 关键证据揭示、人物受伤、伦理选择和感情转折应收住玩笑
- 不连续堆网络梗，不让所有人物使用同一种吐槽语气

## 🔍 信息与伏笔门禁

动笔前填写：

| 类别 | 本章允许出现 | 本章禁止提前出现 |
|---|---|---|
| 读者新知 | 待填 | 待填 |
| 陈默新知 | 待填 | 待填 |
| 其他人物新知 | 待填 | 待填 |
| 伏笔 | 待填编号或“无” | 待填 |
| 误导 | 待填或“无” | 不得破坏公平性 |

关键结论必须满足“两类独立来源”原则；本章只能埋线索时，不得用叙述者替读者下最终结论。

## ✅ 正文完成检查

- [ ] 实际字数处于章节目标范围，超出时有明确剧情理由
- [ ] 开头承接上一章的时间、地点和未解决问题
- [ ] 本章主任务已经完成或形成明确阶段结果
- [ ] 陈默没有被写成无能、冷酷战神或全知角色
- [ ] 专业概念普通读者可以通过情境理解
- [ ] 笑点没有破坏危险、悬疑和情感段落
- [ ] 章尾钩子来自本章因果，不是强行插入新事故
- [ ] 没有提前泄露作者层秘密
- [ ] 正文章节 JSON 的标题、编号、卷名和目录一致

## 🗂️ 完成后同步记录

同一次改动必须更新：

1. `docs/章节摘要.md`
2. `docs/时间线.md`
3. `docs/人物状态.md`
4. `docs/伏笔表.md`
5. `docs/连续性台账.md`
6. `docs/世界观规则.md`，仅在关键设定实际变化时更新；没有变化也要在连续性台账记录“无”
7. `data/catalog.json` 与 `data/book.json`

在前五项维护档案对应的本章记录附近加入统一机器标记：

```text
<!-- chapter:volume-XX/chapter-XXX -->
```

自动检查会用它确认摘要、时间线、人物状态、伏笔表和连续性台账均已处理；即使本章没有新增伏笔，也要在伏笔表写明“本章检查，无状态变化”并保留标记。

正式章节第一次发布时，还必须删除非正式测试章，并把 `data/book.json` 的 `publicationStage` 从 `planning` 改为 `serializing`。
